# 專案簡報：Discord ROAS Bot

## 執行摘要

本文件概述了「Discord ROAS Bot」的專案簡報，這是一款為 Discord 設計的精密經濟模擬機器人。其核心概念是創建一個由玩家驅動的持續性虛擬經濟體，使用者可以在其中參與複雜的經濟活動，例如就業、企業所有權和資源管理。我們主要解決的問題是，Discord 平台內部缺乏具有深度、可長期遊玩且引人入勝的經濟模擬遊戲。我們的關鍵價值主張是提供一個高度真實且動態的經濟模型，以促進社群互動、策略制定和創發式玩法(emergent gameplay)，這遠遠超過了目前僅有的簡單賭博或貨幣機器人。我們的目標市場是專注於遊戲、角色扮演和模擬的既有 Discord 社群。

## 問題陳述

目前具有「經濟」功能的 Discord 機器人通常都非常膚淺。它們依賴於簡單的機制，如隨機生成貨幣、賭博或沒有內在價值的基本物品收集。這些系統缺乏深度，導致玩家迅速失去興趣並引發惡性通貨膨脹。使用者之間沒有進程感、所有權或有意義的經濟互動。現有的解決方案未能創造一個用戶行為會產生持久影響的持續性世界。當務之急在於，抓住那些在日常使用的平台上尋求更複雜、更具協作性體驗的大型活躍社群的參與度。現在解決這個問題，能讓我在這個新生的、基於 Discord 的深度整合遊戲市場中站穩腳跟。

## 建議的解決方案

我們將透過一個機器人，建立一個直接整合到 Discord 中的多層次經濟模擬引擎。該解決方案圍繞五個核心資料庫實體進行架構：`Users`（使用者）、`Departments`（代表國家控制的實體）、`Corporations`（玩家擁有的實體）、`Accounts`（通用金融帳本）和 `Transactions`（所有價值交換的原子記錄）。

我們的關鍵差異化因素是：
1.  **多態所有權：** 公司和帳戶可以由個人用戶、其他公司甚至國家部門擁有，從而創造出複雜的所有權結構。
2.  **國家控制的經濟：** 「部門」（例如財政部、中央銀行）作為系統級的經濟代理人，能夠徵稅、發行貨幣（`SYSTEM_GRANT`）和管理公共服務，提供了一個宏觀經濟層面。
3.  **原子交易：** 一個健全的交易系統確保每一筆資金流動都是平衡且可審計的，防止貨幣重複並確保系統完整性。

這個解決方案將透過為創發式玩法提供結構性基礎而取得成功。經濟將根據其參與者的集體決策而演變，從個人就業選擇到企業投資策略，而非預設的腳本事件。我們的高層次願景是在一個瓶子裡創造一個由真實使用者智慧驅動的「虛擬社會」。

## 目標用戶

### 主要用戶群：策略玩家與社群領袖

*   **用戶畫像：** 年齡 18-35 歲，在 Discord 上高度活躍，通常擔任領導角色（伺服器管理員、公會領袖）。他們是策略遊戲、大型多人線上遊戲和經濟模擬遊戲（例如 EVE Online、Factorio、Civilization）的愛好者。
*   **行為模式：** 他們每天在 Discord 上花費數小時，組織活動和管理社群。他們喜歡可以精通的長期專案和複雜系統。
*   **需求與痛點：** 他們需要更深入的互動工具來保持社群活躍。他們對那些沒有真正策略深度或協作潛力的淺薄機器人感到失望。
*   **目標：** 建立一個蓬勃發展、活躍且獨特的社群，提供超越簡單聊天的持續且引人入勝的體驗。

### 次要用戶群：休閒角色扮演玩家與投入的社群成員

*   **用戶畫像：** 主要用戶群社群的成員。他們喜歡參加團體活動，並正在尋找在社群框架內貢獻和進步的方式。
*   **行為模式：** 他們是伺服器生活的積極參與者，聽從社群領袖的領導，並享受進程感和認可感。
*   **需求與痛點：** 他們希望感覺自己是社群中有價值的一員，並有明確的參與和成就途徑。
*   **目標：** 在遊戲的背景下獲得樂趣、社交並實現個人目標（例如，「賺取」足夠的錢來購買特殊身份組，或創辦一家小「公司」）。

## 目標與成功指標

### 商業目標

*   在推出後的 6 個月內，被 50 個活躍的 Discord 伺服器（成員數 >500）採用。
*   維持機器人及其後端服務 99.9% 的正常運行時間。
*   在第一年年底前，為未來的加值/貨幣化功能建立框架。

### 用戶成功指標

*   如果使用者在第一個月內加入或創建了一家公司，則視為成功。
*   如果一個伺服器中至少有 20% 的成員創建了帳戶並完成至少一筆交易，則視為成功。
*   高留存率，以每週至少執行一次經濟活動的使用者百分比來衡量。

### 關鍵績效指標 (KPIs)

*   **DAU/MAU：** 每日活躍用戶數 / 每月活躍用戶數。
*   **交易量：** 每日處理的總交易筆數。
*   **新帳戶創建率：** 新用戶加入經濟體的速度。
*   **機器人延遲：** 所有機器人指令的平均回應時間。

## MVP 範圍

### 核心功能 (必要功能)

*   **使用者引導：** 使用者可以創建個人金融 `Account`。
*   **部門創建：** 伺服器管理員可以創建國家 `Departments`（例如財政部）。
*   **公司管理：** 使用者可以申請創建 `Corporation`，該申請必須由指定的 `Department` 批准。使用者可以管理其公司的員工。
*   **金融帳戶：** `Users`、`Departments` 和 `Corporations` 都擁有各自的 `Accounts`。
*   **交易：**
    *   使用者可以將資金轉移給其他使用者 (`TRANSFER`)。
    *   公司可以向員工支付薪水 (`SALARY`)。
    *   部門可以對公司徵稅 (`TAX`)。
    *   指定的部門可以創造貨幣 (`SYSTEM_GRANT`)。
*   **基本指令：** 用於查詢餘額、查看交易、轉移資金、管理公司等的 Discord 指令。

### MVP 範圍之外

*   公司間的交易與合約。
*   玩家對玩家的物品交易。
*   股票市場或複雜的金融工具。
*   自動化的生產鏈或資源採集。
*   圖形使用者介面或網路儀表板。

### MVP 成功標準

如果我們能在一個測試伺服器中展示一個穩定、封閉循環的經濟體，MVP 就被視為成功。這意味著使用者可以賺錢（薪水）、花錢（轉帳），而系統的核心實體（公司、部門）可以進行金融互動（稅收、補助），且在 10-20 名使用者的測試組積極使用 48 小時後，不會產生經濟失衡或嚴重錯誤。

## MVP 後的願景

### 第二階段功能

*   **資產與物業管理：** 允許公司和使用者擁有有形資產（例如，「土地」、「工廠」）。
*   **資源與生產系統：** 引入原材料和生產鏈，實現商品的製造和貿易。
*   **市場：** 一個供使用者和公司買賣物品和服務的中央樞紐。

### 長期願景

將該機器人發展成為一個成熟的「虛擬經濟的 Discord 操作系統」。這包括為其他開發者提供廣泛的 API，以建構專門的模組（例如，一個從財政部支付獎勵的「任務」模組，或一個公司可以贊助玩家的「副本」模組）。我們設想的未來是，整個遊戲體驗都將建立在我們的經濟引擎之上。

### 擴展機會

*   **跨伺服器聯盟：** 允許來自不同 Discord 伺服器的公司建立貿易協定或聯盟。
*   **加值功能：** 為超大型伺服器提供進階分析、公司自訂品牌或專用託管服務。
*   **網路儀表板：** 一個功能齊全的網路介面，用於管理財務、公司和查看市場數據。

## 技術考量

### 平台需求

*   **目標平台：** Discord。
*   **瀏覽器/作業系統支援：** MVP 不適用。
*   **效能需求：** 機器人指令回應時間平均應低於 500 毫秒。資料庫必須能夠處理大量交易而不會發生死鎖。

### 技術偏好

*   **前端：** MVP 不適用。
*   **後端：** Python (>=3.12)，使用 `discord.py` 函式庫作為核心框架。
*   **資料庫：** 確認使用 PostgreSQL，透過 `asyncpg` 驅動程式進行存取。
*   **資料庫 ORM/工具集：** 使用 SQLAlchemy 進行資料對應，使用 Alembic 進行資料庫結構遷移。
*   **非同步/效能：** 在非 Windows 平台上將使用 `uvloop` 以提升效能。
*   **託管/基礎設施：** 雲端供應商，如 AWS（RDS for Postgres, EC2/Fargate for the bot）或 Heroku。

### 架構考量

*   **儲存庫結構：** 一個包含 Python 應用程式的單體儲存庫 (monorepo)。
*   **服務架構：** MVP 階段採用單體應用程式即可。它將使用 `discord.py` 和 `SQLAlchemy` 與 Discord API 和 PostgreSQL 資料庫互動。
*   **整合需求：** 唯一外部需求是與 Discord API 直接整合。
*   **安全性/合規性：** 所有資料庫查詢必須參數化以防止 SQL 注入。敏感資料應減至最少，但存取權杖必須安全儲存。

## 限制與假設

### 限制

*   **預算：** 待定，但初期假設將以精簡為主，優先考慮開源和免費層級的雲端服務。
*   **時間表：** MVP 目標在專案啟動後 3 個月內完成。
*   **資源：** 一個小型的開發團隊（1-2 名開發者）。
*   **技術：** 機器人必須在 Discord API 的速率限制內運行。

### 關鍵假設

*   我們假設在 Discord 上存在大量未被滿足的深度經濟模擬需求。
*   我們假設玩家願意學習比一般 Discord 機器人更複雜的系統。
*   我們假設伺服器管理員會將此機器人視為提升社群參與度和留存率的寶貴工具。

## 風險與待解決問題

### 關鍵風險

*   **過於複雜：** 系統可能對休閒使用者來說過於複雜，從而阻礙其被廣泛採用。緩解措施：提供一個非常清晰和引導性的上手流程。
*   **經濟失衡：** 一個調整不當的參數（例如稅率、初始貨幣供給）可能導致快速的通貨膨脹或通貨緊縮。緩解措施：經濟體在推出後將受到密切監控，並將透過部門行動來穩定經濟。
*   **可擴展性：** 一個受歡迎的伺服器可能會產生數百萬筆交易，對資料庫造成壓力。緩解措施：仔細的資料庫結構設計（已完成）、索引以及可擴展託管的規劃。

### 待解決問題

*   稅率、薪水和系統補助金的最佳初始值是多少？
*   如何管理公司批准以防止垃圾訊息或濫用？
*   最有效的使用者上手流程是什麼，既能解釋核心概念又不會讓人感到不知所措？

### 需要進一步研究的領域

*   用於平衡虛擬經濟的博弈論和經濟建模。
*   在 Discord 上為複雜的、基於文本的介面進行使用者體驗（UX）設計。

## 附錄

### A. 研究摘要

該專案基於先前的討論，分析了現有 Discord 經濟機器人的缺點，並設計了一個強大、關聯式的資料庫結構，能夠支援經濟代理人之間複雜的多態關係。核心設計被認為是健全且靈活的。
